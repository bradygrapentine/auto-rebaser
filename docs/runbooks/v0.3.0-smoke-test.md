# v0.3.0 Smoke Test
_2026-05-06_

15-minute manual test plan covering the risky paths in Phase 4 — Device Flow, token storage migration, installation gating, account-switching. Does NOT cover GHES (skip unless you have a GHES instance) or token-expiry refresh (would require 8h wall time).

---

## Prep (1 min)

```bash
git checkout main && git pull
npm run build
```

Open `chrome://extensions`, enable Developer Mode, **Remove** any existing Auto Rebaser install, then **Load unpacked** and pick `<repo>/dist`. Note the extension ID — it should match the `key` field in `manifest.json`.

> **Heads up**: removing the install wipes `storage.local` AND `storage.sync` for this extension. If you want to test the legacy-PAT migration, see §6 first.

---

## 1. Device Flow end-to-end (3 min)

1. Click the toolbar icon. SignInView shows two buttons.
2. Click **sign in with GitHub App (recommended)**.
3. Verify a 2-second-or-less display of an 8-character code (e.g. `ABCD-1234`) with a **copy** button.
4. Verify a new tab opened to `https://github.com/login/device`.
5. Paste / type the code, click **Authorize Auto Rebaser**.
6. Within ~10 seconds, the popup should transition to the PR list.

**Pass**: PR list loads, your authored open PRs appear.
**Fail**: Capture the popup state, the github.com tab URL, and `chrome://extensions` → "service worker" → console errors.

## 2. Auth method surfacing (1 min)

Pop the toolbar icon and verify:

- Footer line shows `via GitHub App on <your-login>` (and any orgs you installed on, comma-separated).
- Open settings (gear icon). New `account` section shows `auth_method: GitHub App` and a `switch to PAT` button.

**Fail**: footer line missing or shows `via PAT` despite signing in via Device Flow → check storage at `chrome://extensions` → "service worker" → Application → IndexedDB / Storage → look for the `auth` key under `chrome.storage.local`.

## 3. Installation coverage display (2 min)

Still on the PR list, verify:

- Repo groups whose owner IS in your installations list have no extra badge.
- Repo groups whose owner ISN'T (e.g. a personal repo when you only installed on an org) show an **App not installed** badge in the group header.
- If the App is installed nowhere accessible to you, the popup shows the install/request CTA above the PR list.

**Note**: the orchestrator will skip rebases on suspended-org repos but you can't easily reach that state without admin access. Skip if you can't easily produce one.

## 4. Switch method PAT → App (2 min)

1. Sign out (settings → account → `switch to PAT` OR header `log-out`).
2. SignInView returns. Click **use a personal access token (legacy)**.
3. Paste a PAT (any valid token with `repo` scope).
4. Verify popup transitions to PR list, footer says `via PAT`.
5. **Migration banner**: verify the banner appears at the top of the PR list ("Your PAT works fine, but GitHub App auth is more secure…"). Click **dismiss**. Reopen popup — banner should stay dismissed.

## 5. 2.9 notifications-scope CTA branching (1 min)

Settings → automations → expand `dismiss stale PR notifications`. Toggle the section ON.

- **PAT user**: should see the existing `grant notifications scope` button.
- **App user**: should see a muted message "Notification cleanup is unavailable when signed in via GitHub App. Switch to PAT…"

Toggle through both auth methods to verify both branches render.

## 6. Legacy PAT migration (5 min — optional but valuable)

Tests the `sync.github_token` → `local.auth` migration path. Requires a v0.1.0–v0.2.1 build of the extension OR manually planting a sync key.

**Easier path** — manually plant the legacy key:

1. While signed out (no `auth` in `storage.local`), open the extension's service worker console (`chrome://extensions` → "service worker").
2. Run:
   ```js
   chrome.storage.sync.set({ github_token: 'ghp_<your-test-pat>' })
   ```
3. Reopen the popup. The first call to `getAuth()` should:
   - Return `{ method: 'pat', token: 'ghp_…' }` to the popup.
   - Write the same shape to `chrome.storage.local.auth`.
   - REMOVE the legacy key from `storage.sync`.

**Verify** in the service-worker console:
```js
chrome.storage.local.get('auth').then(r => console.log(r));
chrome.storage.sync.get('github_token').then(r => console.log(r)); // should be {}
```

Then sign back in normally, verify popup shows the PR list.

## 7. Sanity: nothing broken from prior phases (1 min)

- Click **poll now** in the header. The spinner runs, footer time updates.
- Open settings → automations and confirm no JS errors in the popup console.
- Toggle one automation (e.g. auto-delete merged branches OFF, then ON). Reopen popup and verify the toggle persisted.

## Reporting back

For each failure, capture:
1. Step number from this doc.
2. What you saw vs what was expected.
3. Service worker console output (`chrome://extensions` → "service worker" → click the link).
4. Popup console output (right-click popup → Inspect → Console).
5. Storage state if relevant (`chrome.storage.local.get(null)` and `chrome.storage.sync.get(null)`).

Paste those into a single message and we'll triage.

## What this test does NOT cover

- **Token refresh near expiry** — fires once per ~8 hours of access-token lifetime. Surfaces only on a long-running install. We rely on the unit test for the refresh mechanics + the in-flight-coalescing test.
- **Refresh-token expiry (~6 months)** — same reason, plus the spec says forcing it requires modifying GitHub state.
- **GHES (4.6)** — needs a real GHES instance + a GitHub App registered there. Defer until validating against an actual customer environment.
- **Suspended-installation gating** — requires an org admin to suspend the App while signed in. Cosmetic state; the test plan above just confirms the badge surfaces if such an installation exists.

## Pass criteria for "ready for Web Store submission"

- §1, §2, §3 (visible installations only), §4, §5, §7 — all green.
- §6 — green OR you can articulate why the migration path is safe to skip (e.g. "this build only ships to new users, no upgrades exist").
