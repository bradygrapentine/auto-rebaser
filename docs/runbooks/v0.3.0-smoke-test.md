# v0.3.0 Smoke Test
_2026-05-06_

~25-minute manual test plan covering the risky paths in Phase 4 — Device Flow, token storage migration, installation gating, account-switching, and GHES setup. Does NOT cover token-expiry refresh (would require 8h wall time).

---

## Prep (1 min)

```bash
git checkout main && git pull
npm run build
```

Open `chrome://extensions`, enable Developer Mode, **Remove** any existing Auto Rebaser install, then **Load unpacked** and pick `<repo>/dist`. Note the extension ID — it should match the `key` field in `manifest.json`.

> **Heads up**: removing the install wipes `storage.local` AND `storage.sync` for this extension. If you want to test the legacy-PAT migration, see §6 first.

---

## 1. Device Flow end-to-end (3 min)

1. Click the toolbar icon. SignInView shows two buttons.
2. Click **sign in with GitHub App (recommended)**.
3. The popup transitions to the device-code view: an 8-character code (e.g. `ABCD-1234`) with a `[copy]` button beneath the heading and a green `[copy code & open verification page]` button below it. The popup does NOT auto-open the verification tab — that would steal focus and wipe the code from view.
4. Click `[copy code & open verification page]`. The popup closes (Chrome MV3 behavior on tab focus shift) and a new tab loads `https://github.com/login/device` with the code already in your clipboard.
5. Paste the code, click **Authorize Auto Rebaser**.
6. Click the extension icon again — the popup re-opens directly into the device-code view (resume-on-mount queries the service worker for in-flight flow status). Within ~10 seconds the popup transitions to the PR list automatically.

**Pass**: PR list loads, your authored open PRs appear.
**Fail**: Capture the popup state, the github.com tab URL, and `chrome://extensions` → "service worker" → console errors.

## 2. Auth method surfacing (1 min)

Pop the toolbar icon and verify:

- Footer line shows `via GitHub App on <your-login>` (and any orgs you installed on, comma-separated).
- Open settings (gear icon). New `account` section shows `auth_method: GitHub App` and a `switch to PAT` button.

**Fail**: footer line missing or shows `via PAT` despite signing in via Device Flow → check storage at `chrome://extensions` → "service worker" → Application → IndexedDB / Storage → look for the `auth` key under `chrome.storage.local`.

## 3. Installation coverage display (2 min)

Still on the PR list, verify:

- Repo groups whose owner IS in your installations list have no extra badge.
- Repo groups whose owner ISN'T (e.g. a personal repo when you only installed on an org) show an **App not installed** badge in the group header.
- If the App is installed nowhere accessible to you, the popup shows the install/request CTA above the PR list.

**Note**: the orchestrator will skip rebases on suspended-org repos but you can't easily reach that state without admin access. Skip if you can't easily produce one.

## 4. Switch method, both directions (3 min)

### 4a. App → PAT
1. Currently signed in via App. Settings → account → click `switch to PAT` (this signs out and returns to SignInView).
2. Click **use a personal access token (legacy)**.
3. Paste a PAT with `repo` scope.
4. Verify popup transitions to PR list, footer says `via PAT`, settings → account row reads `auth_method — PAT (legacy)`.
5. **Migration banner**: verify the banner appears flush against the header ("Your PAT works fine, but GitHub App auth is more secure…"). Click **dismiss**. Reopen popup — banner should stay dismissed for this PAT identity.

### 4b. PAT → App (back)
1. From the PR list, click the migration banner's `[switch to GitHub App]` (or settings → account → `switch to GitHub App` if you already dismissed the banner).
2. Verify the popup returns to SignInView and the App / PAT choice is shown.
3. Click **sign in with GitHub App**, walk Device Flow again.
4. After authorisation, verify:
   - Footer shows `via GitHub App on …`.
   - Settings → account reads `auth_method — GitHub App`.
   - The migration banner is GONE (App-mode users never see it).
   - The previously-stored PR cache survives the swap (PR list does NOT reset to empty before re-poll).

**Fail**: if signing back in via App leaves stale PAT data behind, run `chrome.storage.local.get('auth')` and confirm `method === 'github_app'` plus `accessToken` is fresh.

## 5. 2.9 notifications-scope CTA branching (1 min)

Settings → automations → expand `dismiss stale PR notifications`. Toggle the section ON.

- **PAT user**: should see the existing `grant notifications scope` button.
- **App user**: should see a muted message "Notification cleanup is unavailable when signed in via GitHub App. Switch to PAT…"

Toggle through both auth methods to verify both branches render.

## 6. Legacy PAT migration (5 min — optional but valuable)

Tests the `sync.github_token` → `local.auth` migration path. Requires a v0.1.0–v0.2.1 build of the extension OR manually planting a sync key.

**Easier path** — manually plant the legacy key:

1. While signed out (no `auth` in `storage.local`), open the extension's service worker console (`chrome://extensions` → "service worker").
2. Run:
   ```js
   chrome.storage.sync.set({ github_token: 'ghp_<your-test-pat>' })
   ```
3. Reopen the popup. The first call to `getAuth()` should:
   - Return `{ method: 'pat', token: 'ghp_…' }` to the popup.
   - Write the same shape to `chrome.storage.local.auth`.
   - REMOVE the legacy key from `storage.sync`.

**Verify** in the service-worker console:
```js
chrome.storage.local.get('auth').then(r => console.log(r));
chrome.storage.sync.get('github_token').then(r => console.log(r)); // should be {}
```

Then sign back in normally, verify popup shows the PR list.

## 7. GitHub Enterprise Server setup (8 min — only if you have a GHES instance)

Skip if you can't access a GHES host. The end-to-end path is identical to cloud GitHub but routes against your enterprise host instead of `api.github.com`.

### 7a. Pre-reqs (one-time, on the GHES side)

1. On your GHES host, register a new GitHub App (Settings → Developer settings → GitHub Apps → New).
2. App settings:
   - **Homepage URL**: anything you control (e.g. the Auto Rebaser repo URL).
   - **Webhook**: disable.
   - **Permissions**: Repository → Contents (read/write), Pull requests (read/write), Metadata (read). Account → none.
   - **Where can this GitHub App be installed?**: Only on this account (or any installations option that fits your org policy).
3. After saving, note the **Client ID** (looks like `Iv23li…`) — Auto Rebaser needs this to start Device Flow against your host.
4. Install the App on at least one user account or org you can test against.
5. Generate a test PAT (`repo` scope) on the same GHES host as a fallback for §7c.

### 7b. Configure the extension

1. Open the popup (signed out OR signed-in cloud — either is fine; the host change forces a re-auth).
2. Settings → enterprise.
3. Enter the GHES host (no scheme, no path — e.g. `github.acme.corp`). The `[apply]` button hosts a `chrome.permissions.request` for `https://<host>/*` — Chrome will prompt; **accept** it.
4. Once the host is set, the `github_app_client_id` field appears below it. Paste the Client ID from §7a.3.
5. Click `[apply]` again. The hint "Switching hosts requires sign-out + sign-in" applies — sign out from the header.

**Verify**:
- `chrome.storage.sync.get('settings')` shows `enterpriseHost: '<your-host>'` and `enterpriseClientId: 'Iv23li…'`.
- `chrome.permissions.getAll()` lists `https://<your-host>/*` in `origins`.

### 7c. End-to-end against GHES

1. From SignInView, click **sign in with GitHub App**.
2. Confirm the device-code page that opens is `https://<your-ghes-host>/login/device` (NOT `github.com`). Authorize.
3. PR list loads from `<your-ghes-host>/api/v3`. Confirm via service-worker console (`chrome://extensions` → "service worker" → Network tab) that REST + GraphQL calls hit the GHES host.
4. Footer shows `via GitHub App on …` exactly as cloud — no special GHES badge expected.
5. **PAT fallback**: sign out, choose **use a personal access token**, paste the GHES PAT. Verify the PR list still loads (PAT path uses the same host config).

### 7d. Tear-down — return to cloud

1. Settings → enterprise → clear the `github_host` field, click `[apply]`.
2. Confirm Chrome dialog if it asks to remove the host permission.
3. Verify `chrome.permissions.getAll().origins` no longer lists the GHES host (the optional permission is auto-removed).
4. Sign out + sign back in via the cloud Device Flow.

**Fail modes**:
- Device Flow opens to `github.com` instead of GHES → `enterpriseClientId` not saved, or host validation rejected the entry. Check the host-error inline alert.
- 404 on REST calls → host is set but missing `/api/v3` in the request path. That's a code bug; capture the failing URL.
- Permission prompt never appears on `[apply]` → the `optional_host_permissions` declaration in `manifest.json` is missing. Wouldn't pass build typecheck, but worth confirming if everything else looks right.

## 8. Sanity: nothing broken from prior phases (1 min)

- Click **poll now** in the header. The spinner runs, footer time updates.
- Open settings → automations and confirm no JS errors in the popup console.
- Toggle one automation (e.g. auto-delete merged branches OFF, then ON). Reopen popup and verify the toggle persisted.

## What this test does NOT cover

- **Token refresh near expiry** — fires once per ~8 hours of access-token lifetime. Surfaces only on a long-running install. We rely on the unit test for the refresh mechanics + the in-flight-coalescing test.
- **Refresh-token expiry (~6 months)** — same reason, plus the spec says forcing it requires modifying GitHub state.
- **Suspended-installation gating** — requires an org admin to suspend the App while signed in. Cosmetic state; the test plan above just confirms the badge surfaces if such an installation exists.

## Pass criteria for "ready for Web Store submission"

- §1, §2, §3 (visible installations only), §4 (both directions), §5, §8 — all green.
- §6 — green OR you can articulate why the migration path is safe to skip (e.g. "this build only ships to new users, no upgrades exist").
- §7 — green if you have a GHES instance to test against; otherwise document "deferred — no test environment" and note that §7 will run before any GHES customer is onboarded.
